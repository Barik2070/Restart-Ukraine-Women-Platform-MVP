# Restart Ukraine Women Platform — Design System

Документ описує структуру Figma-файлу для дизайнера. Реальний Figma-файл не створюється в репозиторії; цей файл є handoff-специфікацією для підготовки дизайн-системи та макетів MVP.

Назва Figma-файлу: **Restart Ukraine Women Platform — Design System**

## 1. Мета Figma-файлу

Figma-файл має бути єдиним джерелом візуальних правил для MVP Restart Ukraine Women Platform: бренду, кольорів, типографіки, компонентів, сторінок, мобільних адаптацій і прототипних зв'язків.

Дизайн має підтримувати довіру, ясність і практичність. Платформа не повинна виглядати як благодійна кампанія про вразливість, а як професійний сервіс повернення в Україну з планом, навігацією, навчальними напрямами та партнерською підтримкою.

## 2. Структура Pages у Figma

### 00 Cover

Обкладинка дизайн-системи.

Потрібно розмістити:

- назву файлу: **Restart Ukraine Women Platform — Design System**;
- назву продукту в інтерфейсі: **Restart Ukraine Women**;
- слоган: **Повернення в Україну з планом**;
- короткий опис MVP;
- дату або версію дизайн-системи;
- посилання на основні секції: Brand, Colors, Typography, Components, Prototype.

### 01 Brand

Сторінка для базових brand rules.

Потрібно описати:

- назву в інтерфейсі: **Restart Ukraine Women**;
- слоган: **Повернення в Україну з планом**;
- тон: спокійний, впевнений, практичний, людяний;
- стиль: сучасний, світлий, професійний, європейський, український без надмірної символіки;
- візуальний принцип: менше декоративності, більше ясної структури, простору, читабельності та довіри;
- заборона на образ жінки як жертви.

Рекомендований настрій: жінка як активна учасниця рішення, фахівчиня, підприємиця, майбутня водійка, фермерка, операторка AI-інструментів або власниця малого бізнесу.

### 02 Colors

Сторінка з кольоровими стилями та прикладами використання.

Основна палітра:

| Token | HEX | Використання |
| --- | --- | --- |
| Deep blue | `#1E3A8A` | header, основні акценти, заголовки, primary CTA |
| Warm yellow | `#FBBF24` | акценти, highlight, важливі CTA, маркери прогресу |
| Warm beige | `#F8F4EC` | світлі фони секцій, м'які інформаційні блоки |
| White | `#FFFFFF` | основний фон, картки, форми |
| Graphite | `#1F2937` | основний текст, footer, важливі підписи |
| Soft green | `#6BAA75` | success states, результати, позитивні KPI |
| Light gray | `#E5E7EB` | borders, dividers, disabled states, input outlines |

Правила використання:

- deep blue має бути головним професійним кольором, але не повинен затемнювати весь інтерфейс;
- warm yellow використовується дозовано, як теплий український акцент;
- warm beige застосовується для спокійних секцій і не має перетворювати інтерфейс на монотонно-бежевий;
- graphite використовується для максимальної читабельності текстів;
- soft green застосовується для результатів, успішних дій і позитивних статусів;
- light gray потрібен для нейтральної структури, меж і inactive states.

### 03 Typography

Сторінка з типографічними стилями для desktop і mobile.

Шрифти:

- headings: **Manrope**;
- body: **Inter**;
- buttons: **Inter Medium** або **Manrope Medium**.

Рекомендована ієрархія:

| Style | Font | Призначення |
| --- | --- | --- |
| H1 | Manrope Bold | головний заголовок сторінки |
| H2 | Manrope SemiBold | заголовки великих секцій |
| H3 | Manrope SemiBold | заголовки карток і підсекцій |
| Body | Inter Regular | основний текст |
| Body medium | Inter Medium | важливі фрагменти, labels |
| Small | Inter Regular | підписи, допоміжні тексти |
| Button | Inter Medium або Manrope Medium | кнопки та CTA |

Правила:

- заголовки мають бути короткими й читабельними;
- body text не має бути дрібним, особливо у формах;
- на mobile заголовки зменшуються, але зберігають чітку ієрархію;
- кнопки мають мати однакову вагу шрифту та достатній tap target.

### 04 Components

Сторінка з компонентами, варіантами, станами та mobile rules. Кожен компонент має бути оформлений як reusable component з варіантами, якщо це доречно.

#### Header

Призначення: головна навігація сайту, доступ до програм, донорської сторінки, контактів і анкети.

Стани: default, sticky або scrolled, active page, mobile open, mobile closed.

Де використовується: усі публічні сторінки MVP.

Mobile: логотип/назва зліва, menu icon справа, навігація відкривається як зрозуміле вертикальне меню; CTA до анкети має залишатися доступним.

#### Footer

Призначення: завершення сторінки, службові посилання, контакти, юридична інформація, партнерські посилання.

Стани: default.

Де використовується: усі публічні сторінки.

Mobile: колонки перетворюються на вертикальні блоки, текст не має бути надто дрібним, контактні дії мають легко натискатися.

#### Primary button

Призначення: головна дія користувачки або партнера.

Стани: default, hover, active, focus, disabled, loading.

Де використовується: hero, CTA sections, forms, application flow, donor page.

Mobile: мінімальна висота 44 px, достатній горизонтальний padding, текст не обрізається.

#### Secondary button

Призначення: додаткова дія без конкуренції з primary CTA.

Стани: default, hover, active, focus, disabled.

Де використовується: переходи до деталей програм, завантаження документів, другорядні сторінкові дії.

Mobile: може займати повну ширину, якщо поруч немає місця для двох кнопок.

#### CTA section

Призначення: чітко вести користувачку до анкети або партнера до цільової дії.

Стани: default, with secondary action, donor variant, program variant.

Де використовується: homepage, program pages, donor page, contacts.

Mobile: заголовок, короткий текст і кнопки мають складатися вертикально; primary CTA не губиться внизу блоку.

#### Program card

Призначення: коротко презентувати напрям програми.

Стани: default, hover, active або selected, unavailable/future.

Де використовується: homepage, programs page, program detail pages.

Mobile: картки йдуть в одну колонку; іконка, назва, опис і CTA мають читатися без горизонтального скролу.

#### Result card

Призначення: показати очікуваний результат для учасниці після проходження програми.

Стани: default, highlighted.

Де використовується: homepage, program pages, donor materials.

Mobile: короткі формулювання, без перевантаження числовими або декоративними елементами.

#### Donor KPI card

Призначення: показати ключові показники для донорів і партнерів.

Стани: default, positive change, neutral, placeholder/MVP.

Де використовується: donor page, partner materials, impact sections.

Mobile: KPI має залишатися великим і зрозумілим; підпис переноситься на новий рядок без обрізання.

#### Step timeline

Призначення: пояснити послідовність шляху користувачки: анкета, відбір, навчання, підтримка, результат.

Стани: default, current step, completed step.

Де використовується: homepage, application form, program pages.

Mobile: timeline стає вертикальною; номери кроків і назви мають бути помітними.

#### Form input

Призначення: введення текстових даних в анкеті або контактній формі.

Стани: empty, filled, focus, error, disabled, success.

Де використовується: application form, contacts, CRM-related flows.

Mobile: label завжди видимий; error text розміщується під полем; input має бути зручним для touch.

#### Select

Призначення: вибір одного значення зі списку.

Стани: closed, open, selected, focus, error, disabled.

Де використовується: application form, фільтри програм, CRM/admin scenarios.

Mobile: список має бути читабельним, без дрібних пунктів; довгі варіанти переносяться.

#### Checkbox

Призначення: підтвердження згоди або вибір кількох опцій.

Стани: unchecked, checked, focus, error, disabled.

Де використовується: application form, privacy consent, program interests.

Mobile: tap area має включати label; текст згоди не має бути занадто дрібним.

#### Radio button

Призначення: вибір одного варіанту з групи.

Стани: unselected, selected, focus, error, disabled.

Де використовується: application form, вибір напряму, опитування готовності.

Mobile: варіанти розміщуються вертикально; label має бути повністю видимим.

#### Success message

Призначення: підтвердити успішну дію, наприклад відправлення анкети.

Стани: default, with next steps, with contact info.

Де використовується: application form, contact form, партнерські дії.

Mobile: повідомлення має бути коротким, з чіткою наступною дією.

#### FAQ accordion

Призначення: компактно відповідати на часті питання.

Стани: collapsed, expanded, hover, focus.

Де використовується: homepage, program pages, donor page, application form.

Mobile: один відкритий пункт не повинен створювати плутанину; touch target для заголовка достатній.

#### Mobile menu

Призначення: мобільна навігація сайту.

Стани: closed, open, active item, with CTA.

Де використовується: mobile version усіх сторінок.

Mobile: меню має мати чітку ієрархію, достатні відступи, зрозумілу кнопку закриття і видимий CTA.

#### Contact card

Призначення: показати канал зв'язку або відповідальну контактну точку.

Стани: default, clickable, highlighted.

Де використовується: contacts page, footer, donor page, program pages.

Mobile: телефони, email і Telegram мають бути натискними; картки йдуть вертикально.

#### Document download card

Призначення: дати доступ до документів, презентацій, політик або матеріалів для партнерів.

Стани: default, hover, disabled/unavailable, loading.

Де використовується: donor page, partner page, methodology, privacy-related pages.

Mobile: назва документа, формат і CTA мають бути видимими без горизонтального скролу.

### 05 Homepage

Сторінка для макетів головної сторінки.

Потрібно підготувати:

- desktop layout;
- tablet або medium breakpoint за потреби;
- mobile layout;
- hero з назвою Restart Ukraine Women і слоганом;
- блок програм;
- блок результатів;
- короткий шлях користувачки;
- CTA до `application.html`;
- партнерський або донорський блок без перевантаження.

### 06 Program Pages

Сторінка для напрямів програм.

Потрібно підготувати макети для:

- автонапряму;
- агронапряму;
- бізнес-напряму;
- AI-напряму.

Вимоги:

- кожен напрям має мати зрозумілий практичний результат;
- AI-модуль має виглядати практично, не технічно складно;
- агронапрям має виглядати сучасно, не архаїчно;
- CTA веде до `application.html`;
- mobile version обов'язкова.

### 07 Application Form

Сторінка для анкети та станів форми.

Потрібно підготувати:

- form layout;
- required fields;
- validation errors;
- consent checkbox;
- success state;
- mobile version;
- empty, filled, error і submitted стани.

Форма має виглядати простою, дружньою і безпечною. Вона не повинна створювати відчуття складної бюрократичної заявки.

### 08 Donor Page

Сторінка для донорської та партнерської комунікації.

Потрібно підготувати:

- hero для донорів;
- impact/KPI cards;
- опис програми як професійної ініціативи;
- document download cards;
- contact CTA;
- mobile version.

Сторінка має виглядати професійно, структуровано і достатньо стримано для європейських донорів та партнерів.

### 09 Mobile

Окрема сторінка для mobile-first перевірки.

Потрібно зібрати:

- mobile header;
- mobile menu;
- mobile homepage;
- mobile program page;
- mobile application form;
- mobile donor page;
- типові компоненти в мобільному вигляді.

Mobile version є обов'язковою частиною handoff, а не додатковим nice-to-have.

### 10 Prototype

Сторінка для прототипних зв'язків.

Потрібно налаштувати основні переходи:

- Homepage → Application Form;
- Homepage → Program Pages;
- Program Pages → Application Form;
- Donor Page → партнерська дія або контакт;
- Header navigation між ключовими сторінками;
- Mobile menu navigation;
- Application Form → Success Message.

## 3. Handoff rules

1. Дизайнер не змінює сенс текстів без погодження.
2. Усі компоненти мають бути адаптивними.
3. Усі CTA мають вести до `application.html` або партнерської дії.
4. Mobile version обов'язкова.
5. Донорська сторінка має виглядати професійно.
6. AI-модуль має виглядати практично, не технічно складно.
7. Агронапрям має виглядати сучасно.
8. Не використовувати образ жінки як жертви.

## 4. Фінальна перевірка перед handoff

Перед передачею дизайну в розробку потрібно перевірити:

- усі pages створені та названі згідно зі структурою;
- color styles і text styles оформлені як reusable styles;
- компоненти мають states і mobile variants;
- CTA-переходи відповідають правилам;
- mobile макети не містять обрізаного тексту або занадто дрібних tap targets;
- донорська сторінка виглядає як професійний impact/product page;
- візуальна мова підтримує гідність, дію і повернення з планом.
